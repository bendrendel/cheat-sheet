APIs

Basics

  • API stands for Application Programming Interface
  • Used in programming languages to abstract away details and provide easier to use access to functionality
  • Provides specific rules to outside applications for how to interact with the data, like how to retrieve data that the outside application can then use
  • Client-side javascript APIs fall into two categories
    • Browser APIs are APIs built into web browser that provide access to functionality written in lower level languages like C or Rust. Generally provide functionality through an object made available in the browser environment.
    • Third-Party APIs are APIs not built into the web browser. On the web, often a layer of software on a server that handles operations on a database. Often data is returned as a JSON object, which you can manipulate to use in your application
      • Library/Framework APIs: These are third-party apis that are imported by linking the src url in your html file using a script tag with the src set to the third party api url. Then you can use objects in the linked code. There may be css to link to as well using a link tag with the href set to the third-party api's css.
      • REST APIs: These require you to make an HTTP request to a server to retrieve data. Oten you send URL parameters to the API's URL, and it sends back data based on that, most often a JSON response. Nice thing is that it works just like requesting a webpage from a server with the REST HTTP model.

Browser APIs

  • Manipulating documents, like the DOM API
  • Fetching data from a server, like XMLHttpRequest, and Fetch
  • Drawing and manipulating graphics, like Canvas and WebGL
  • Audio and Video, like HTMLMediaElement, Web Audio, and WebRTC
  • Devices, like notifications API, or vibration API
  • Client-side storage, like Web Storage, and IndexedDB

Third-Party APIs

  • Free APIs: Some APIs like Yelp offer their API to the public to expand their reach because they make money the more their data gets viewed
  • Paid APIs: Paid APIs like twilio offer valuable services to programmers for a monthly fee or per use fee
  • Internal APIs: Developers will use APIs in their apps to allow for easy expansion, so different versions of their software (web, mobile, etc.) can easily use the API to interact with the data

API Documentation

  • APIs will provide documentation of how to utilize the API
  • API Key this is often require to be registered with a third-party API to use it, it should be kept secret because malicious users could use it to rack up charges on paid APIs

Naming Resources

  • In REST, everything that can be a target of a hypertext reference is a resource, and these resources are found at particular URIs. Creating these URIs should follow an easy to use naming convention that is consistent across the API. Forward slashes (/) are used to indicate heirarchical relationships. Don't use forward slashes at the end (trailing) of the URI and also never put file extensions at the end of the URI, these add no semantic value (file type can be seen in the http headers). Hyphens should be used to separate words if clarity is needed, and underscores should never be used because certain fonts obscure them. Always use lower case letters only. Don't put CRUD actions in the URI, this can be found in the type of request (GET, PUT, etc). /li>
  • Resources can either be collections like /customers or singletons like /customers/{customerId}
  • Resources can contain sub-collections which can have singletons as well like /customers/{customerId}/accounts or /customers/{customerId}/accounts/{accountId}
  • Use nouns instead of verbs to name resources, except controller resources
  • Use query parameters to filter or sort results, instead of a new URI, like http://api.example.com/device-management/managed-devices?region=USA&brand=XYZ&sort=installation-date
  • Resources are categorized as follows:
    • document: this is a singleton resource contained in a collection, like http://api.example.com/device-management/managed-devices/{device-id} or http://api.example.com/user-management/users/admin
    • collection: this is a server-managed directory of resources, which would control what users can add/delete from the collection, like http://api.example.com/device-management/managed-devices or http://api.example.com/user-management/users/{id}/accounts
    • store: this is a client-managed repository of resources, where each stored resource has a URI chosen by the client when originally added, like http://api.example.com/song-management/users/{id}/playlists
    • controller: this is a resource that is procedural like functions with inputs and outputs and should use a verb, like http://api.example.com/cart-management/users/{id}/cart/checkout or http://api.example.com/song-management/users/{id}/playlist/play
