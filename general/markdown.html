Overview

  • Markdown is a text-formatting language designed for the web with PERL
  • Widely used in software/technical documentation, many communication applications like skype, major websites like wordpress, writing blogs posts for a website, to take notes, implemented in emails, and multiple places in GitHub (Gists, comments in issues and pull requests, and files with markdown extension)
  • It is designed to be lighter weight than HTML to be more accessible, but can't do nearly as much as HTML. It is designed as a subset of HTML.
  • The original markdown was designed in 2004, but more flavors have been created since then that incorporate more features from HTML. CommonMark is a major one. Github flavored markdown (GFM) is another one that expands upon commonmark. Trello is another big one that has its own implementation of markdown.
  • Markdown processors or parsers are included in certain software that can read markdown, it converts markdown to HTML
  • File extension is .markdown or .md

Readme Files

Every programming project should include a README.md file, GitHub automatically displays it on the landing page of the repository. It should contain:

  • Title (h1) usually the project name and possibly brief description and good section titles (h2), possibly subsections (h3-h6)
  • Introduction in a few sentences explaining the point of the project, what problems it solves or incentive for creating it. For coding interview projects, explain how requirements have been met.
  • The technologies used in the project, including libraries and versions
  • How to run the project, if there are minimum hardware requirements, how to run in local environment or install it, for web projects it may jsut be a link to the deployed project. For installing, provide step by step instructions like clone from github, install things, etc.
  • Usage, how to use the project maybe with screenshots
  • How to contribute if applicable, possibly would go in its own file
  • credits, authors
  • license

Optionally, include:

  • Table of contents - for extensive documents, linked to headings
  • Illustrations - logo, diagrams, screenshots, etc
  • Scope of functionalities/features - if not obvious
  • Examples of use
  • Project status - especially if currently being developed, explaining planned changes
  • Sources - e.g. if following a tutorial, or inspired by another project
  • Other information - author, contact info, license, how to contribute, for coding interviews could include info on the time/space complexity of alogrithms/data structures used in the problem.

Markdown Syntax

Headers

  • # This will make an h1 tag
  • ###### You can go up to an h6 tag with multiple # signs

Emphasis

  • *Italic text enclosed in one asterisk (or one underscore)*
  • **Bold text eclosed in two asterisks (or two underscores)**
  • _Example of combining italics with **bold**_

Line Breaks and Horizontal Rules

  • Must use a blank line to get a line break between paragraphs
  • Or leave two spaces at end of line to get line break within paragraph
  • Use three or more dashes, asterisks, or underscores to create a horizontal rule

Lists

  • * Bulleted lists begin each item with an asterisk (or a - or + sign)
      * Create a sub-bullet with two spaces and then an asterisk (or a dash)
  • 1. Create numbered lists by simply starting line with number and period
      1a. Here is sub-item for a numbered list

Links

  • [Displayed link name](url "optional tool tip")
  • url could be relative reference to repository file like ../blob/master/LICENSE or on the web like https://google.com
  • url could be link to a heading in the document, where the heading is all lowercase and spaces replaced with dashes, and preceded by a single # sign, e.g. [Section A](#section-a)
  • Instead of url, can use a reference text, often a single number like 1, 2, 3, defined later in document like so:
    • [Optional displayed link name][reference text]
    • later in document:
    • [reference text]: https://google.com

Images

  • ![Alt text](url)
  • url could be relative reference to repository file like ./images/logo.png or on the web
  • Can also use reference method like with links, instead of url

Blockquotes

  • > Start a line with an arrow bracket to create a formatted block quote
  • > Can include multiple lines in the blockquote, handy to emulate email reply text

Inline Code

  • For code snippets (html, css, js, anything) use backticks `const myVar` for special formatting

HTML

  • Can also use raw HTML syntax in a markdown file, and it will generally work

Additional 'Github-flavored Markdown' Syntax

Strikethrough

  • ~~Enclose with two tildes for strikethrough text~~

Emojis

  • Use the codes found here to insert emojis

Task Lists

  • In descriptions and comments of Issues and Pull Requests, and Gist coments and Gist markdown files, you can include a task list
  • - [x] Task lists include a checkbox, this item is checked
  • - [ ] This item will be unchecked
  • - [ ] Can also use ordered lists
  • - [ ] If included in the first comment of an issue, github will show a progress bar in the list of issues

Tables

  • The below markup will be formatted as a table using | for col breaks, at least 3 dashes for header break and colons to align text in columns
  • First Header | Second Header | Third Header
  • ------------ | :-----------: | -----------:
  • Content in row 1 col 1 | Centered content in row 1 col 2 | Right-aligned content
  • Content in row 2 col 1 | Centered content in row 2 col 2 | Right-aligned content

Links

  • Any URL in the text like http://www.github.com/ will automatically be a clickable link
  • In description and comments of Issues and Pull Requests, SHA references like below automatically become a link to the commit on Github
    • 16c999e8c71134401a78d4d46435517b2271d6ac
    • bendrendel@16c999e8c71134401a78d4d46435517b2271d6ac
    • bendrendel/repository-name@16c999e8c71134401a78d4d46435517b2271d6ac
  • In description and comments of Issues and Pull Requests, any number referring to an Issue or Pull Request like below becomes a link
    • #1
    • bendrendel#1
    • bendrendel/repository-name#1

Username @ mentions

  • In description and comments of Issues and Pull Requests, Using @bendrendel or other username will notify the person or team to view the comment

                    return 'something';
                }
